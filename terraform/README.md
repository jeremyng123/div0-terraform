# Terraform — div0 homelab (Proxmox docker LXC + Dockge stacks)

This project reproduces, from scratch:

1. The **`docker` LXC container** on Proxmox node `pve` (unprivileged Debian,
   4 cores / 4 GB RAM / 512 MB swap, 100 GB rootfs on `local-lvm`,
   `nesting`+`keyctl` for Docker-in-LXC, DHCP on `vmbr0`, starts on boot).
2. **Dockge** — the compose-stack manager UI (`:5001`), from
   [`../div0-homelab-stack/dockge`](../div0-homelab-stack/dockge), deployed to
   `/opt/dockge` (outside the managed stacks dir) so it manages every stack below
   without listing itself.
3. The **Dockge compose stacks** in
   [`../div0-homelab-stack/stacks`](../div0-homelab-stack/stacks), deployed into
   `/opt/stacks` with their secrets rendered from Terraform variables:
   - `docker-socket-proxy` — read-only Docker API proxy (`docker-socket-net`)
   - `dozzle` — log viewer, consumes the proxy
   - `technitium` — DNS server (`observability`)
   - `lgtm-stack` — Grafana / Loki / Tempo / Mimir / Pyroscope / Alloy + Technitium exporter

The source repo is laid out so each piece maps to a destination:

```
div0-homelab-stack/
  stacks/   -> /opt/stacks   (DOCKGE_STACKS_DIR — the managed stacks)
  dockge/   -> /opt/dockge   (Dockge itself)
```

## How it works

| Layer | Tool |
|-------|------|
| Provider | `bpg/proxmox` creates the LXC via the PVE API |
| Secrets | sensitive TF vars → `templatefile` → gitignored `.env` files |
| App deploy | `terraform_data` provisioners SSH into the CT, copy `stacks/` + `dockge/` + `.env`, run `bootstrap.sh` |
| Bootstrap | installs Docker if missing, creates the `external` networks, `docker compose up -d` Dockge then each stack in dependency order |

The stack folders keep their original compose layout, so the result is still
**Dockge-compatible** (stacks land in `/opt/stacks/<name>`).

## Prerequisites

- Terraform ≥ 1.5
- A Proxmox **API token** (`Datacenter → Permissions → API Tokens`) with rights
  to create containers.
- An **LXC template** present on the node. List/download:
  ```bash
  pveam available --section system | grep debian-12
  pveam download local debian-12-standard_12.7-1_amd64.tar.zst
  ```
- An **SSH keypair** for the container — handled automatically: Terraform reads
  `./.ssh/proxmox_homelab(.pub)` and, if missing, generates an ed25519 keypair
  there on first apply (the public key is injected for `root`; the private key
  is used by the deploy provisioner). Point `ssh_public_key_path` /
  `ssh_private_key_path` at an existing key to bring your own instead.

## Usage

```bash
cd terraform
cp terraform.tfvars.example terraform.tfvars   # then edit it
terraform init
terraform plan
terraform apply
```

`terraform apply` will: create the LXC → wait for its IP → copy `stacks/` and
`dockge/` in → install Docker → bring Dockge and all four stacks up. Service URLs
are printed as outputs.

### Redeploying after changing compose files or secrets

The deploy re-runs automatically when a `compose.yaml`, a secret, or the
bootstrap script changes. To force it otherwise, bump `deploy_version`:

```bash
terraform apply -var deploy_version=2
```

## Notes & caveats

- **Auth must be root@pam username/password**, not an API token. The container
  sets the `keyctl` feature flag, and Proxmox only allows feature flags other
  than `nesting` to be changed by `root@pam` via a login ticket — API tokens get
  a `403`. Set `proxmox_password` (with `proxmox_username = "root@pam"`). Use a
  token only if you first remove `keyctl` from `main.tf`.

- **VMID is auto-allocated** by default (`vmid` unset → Proxmox picks the next
  free ID: 100, 101, 102, ...), so an existing CT never blocks creation. Set
  `vmid` to pin a specific ID, or `terraform import` to adopt an existing one.
- **MAC address** is auto-generated by default. Only set `mac_address` to the
  original `BC:24:11:D1:FD:7E` if the original container is gone (duplicate MACs
  break networking).
- **External networks**: the compose files mark `docker-socket-net` and
  `observability` as `external`; `bootstrap.sh` creates them first. (The comment
  in the proxy compose mentions `proxy` — the actual network key is
  `docker-socket-net`, which is what gets created.)
- **Alloy** runs `privileged` + `pid: host` and mounts the real Docker socket.
  That's the deliberate exception in your setup and is reproduced as-is; it
  relies on the LXC's `nesting`/`keyctl` features.
- **Dockge** is the stack-manager UI. Its compose lives in
  `../div0-homelab-stack/dockge` and deploys to `/opt/dockge` (`dockge_dest_path`)
  — outside `/opt/stacks` (`stacks_dest_path`) so it does **not** appear as a
  stack in its own UI. It mounts the real Docker socket **read-write** (not the
  read-only proxy) because it shells out to `docker compose`, and binds
  `/opt/stacks` at the *same path* inside the container so the stacks' relative
  bind mounts resolve. `DOCKGE_STACKS_DIR` is hardcoded to `/opt/stacks` in that
  compose; change it there too if you override `stacks_dest_path`.
- Secrets are rendered to `./.rendered/` locally (gitignored) before being
  pushed. `terraform.tfvars` and `*.tfstate` hold secrets — keep them out of git
  (already covered by `.gitignore`).
